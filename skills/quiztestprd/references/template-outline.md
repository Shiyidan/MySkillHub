# QuizTest PRD 标准模板（完整版）

本模板用于将任一功能需求整理为“可测试、可验收”的 PRD。  
请优先回答“功能怎么走得通、谁在什么条件下能触发、状态如何变化、失败时怎么回退”，而不是大量页面布局描述。

---

## 1. 功能边界与目的

### 1.1 目标一句话

用 1 句话说明本功能解决什么问题、服务于哪个角色（非技术用户也能看懂）。

示例：  
- 学生可在首页看到诊断测试入口，完成一次能力评测并获得进阶建议；管理员可统一管理试卷上下线状态。

### 1.2 适用对象与角色边界

列出本功能涉及角色及权限差异（逐项写明）：

- 学生：可执行/不可执行
- 会员：可执行/不可执行
- 管理员：可执行/不可执行

### 1.3 业务输入范围（Scope）

- 本次功能**必须包含**的对象、流程、状态、接口来源  
- 本次不做的内容（非目标）：明确排除，避免范围蔓延

### 1.4 术语与状态定义（建议统一）

写清楚以下状态术语（必须有对应解释）：

- 已下线 / 已发布 / 已上线 / 进行中 / 已完成 / 已过期 / 暂停 / 已取消
- 额度、试卷类型、题目来源、知识点分析状态等核心字段含义

### 1.5 依赖与来源

列出依赖项：

- 来源数据（数据库表 / 缓存 / 文件 / 第三方）
- 关键字段来源（例如：试卷列表是否来自真题库、状态来源于谁）
- 依赖接口与调用时机

### 1.6 验收说明（该节可直接作为测试前提）

- 该功能不满足以下条件则不应展示入口/按钮/结果
- 不满足条件时的替代文案/动作

---

## 2. 页面描述

### 2.1 页面入口与跳转路径

- 页面在什么路径打开（不需要全量 URL，可写“从 XX 菜单进入”）
- 角色和身份是否会影响入口是否可见
- 页面打开后的默认加载动作

### 2.2 页面核心对象与信息

把页面关键对象按“用户可感知/系统可操作”分别列出，例如：

- 列表/表格的每条记录字段含义
- 卡片/弹窗中的提示语
- 操作按钮（启用、禁用、编辑、导入、重试）

### 2.3 交互行为（最少列出）

对用户可见/可点击项逐项说明：

- 按钮文案与动作
- 点击后的直接反馈（弹窗、提示、跳转、加载、禁用）
- 是否受当前状态限制

### 2.4 页面数据与展现节奏

- 加载、空态、错误态、空列表处理方式
- 分页、过滤、排序、搜索是否支持
- 表头/状态文案与排序是否可复用

### 2.5 权限与可见性

- 哪些角色看到哪些字段与操作
- 哪些字段仅管理员可见（或仅后台可见）

---

## 3. 核心流程

> 推荐以“事件 -> 读取 -> 校验 -> 落库/更新 -> 回写 -> 呈现”顺序书写。

### 3.1 触发路径

逐条列“从何触发”：

- 页面进入触发：  
  - 读取哪些列表
  - 使用哪些过滤条件
  - 如何确定初始状态
- 用户操作触发：  
  - 开始/继续 / 停止 / 提交 / 编辑 / 审核 / 上线 / 下线 等
- 异步定时触发（如分析进度/轮询）：
  - 间隔、触发条件、最大时长、最终退出条件

### 3.2 数据处理规则

每个关键动作写明：

- 输入参数（来自表单/选择/当前用户上下文）
- 后端校验项
- 数据落库规则（新增/更新/幂等/冲突处理）
- 列表状态如何变化

### 3.3 状态变化映射（示例）

每项列：`当前状态 -> 点击动作 -> 更新条件 -> 结果状态 -> 影响页面`

### 3.4 审核、权限、来源过滤规则

- 哪些内容默认不显示（如考试类型不匹配、未上线、无版权/无权限）
- 来源标记优先级（真题/仿真/AI）
- 归档或历史数据的处理逻辑

### 3.5 结果与回写

- 成功反馈（成功态文本、列表刷新、跳转）
- 成果数据在后续页面的展示方式（报告、明细、记录）
- 是否支持继续、重试、撤回、重测等动作

### 3.6 核心验收点（建议 5~8 条）

- 列出可直接用于测试验收的操作点，每条一条。

---

## 4. 异常容错

### 4.1 输入类异常

- 参数缺失 / 格式错误 / 权限不足 / 非法状态下操作
- 如何提示用户、如何阻断提交、是否可继续纠正

### 4.2 数据类异常

- 数据源为空 / 接口返回异常 / 跨页分页失效 / 字段缺失
- 是否回退到空态、重试按钮、或只保留可操作项

### 4.3 并发与时序异常

- 重复点击、重复提交、窗口多开
- 进度恢复与重复测试（继续/中断）策略

### 4.4 异步流程异常

- 分析中断、超时、超量加载
- 前端如何展示“处理中/失败/可重试”状态

### 4.5 权限与安全边界

- 登录过期/403/令牌失效
- 用户身份变更导致页面权限收敛时的降级策略

### 4.6 异常验收点（建议 3~6 条）

- 指定每类异常下的用户可见表现和系统恢复动作。

---

## 5. 后续与非目标说明

### 5.1 本次交付包含

- 当前版本必须上线的功能点清单（与上文流程一一对应）
- 与现有模块的兼容性承诺

### 5.2 本次不做（非目标）

- 清晰写出暂不实现项，避免评审时误当已实现功能
- 对“可复用基础功能”与“后续新增功能”做区分

### 5.3 迭代建议（非本次交付）

- 明确下一阶段可接入内容（例如：更细的报表、更多试卷类型、导入预校验提示优化）

---

## 模板输出清单（快速落地）

完成每个文档时至少补齐：

1. 功能边界与目的：一句话目标 + 角色边界 + 包含/排除清单  
2. 页面描述：入口、字段、操作、默认态/空态/错误态  
3. 核心流程：触发事件、字段来源、状态迁移、结果展示  
4. 异常容错：输入异常、数据异常、权限与异步异常  
5. 后续与非目标：本次范围、排除项、后续扩展建议  

每个章节末尾建议附一段“验收可读性说明”，便于直接转为测试用例清单。

## 附：验收映射示例（可拷贝）

- 场景：用户未登录访问受限页面  
  - 前置：未登录 token 失效  
  - 预期：页面回到登录/显示权限提示，并保留重试入口  

- 场景：列表无可见数据  
  - 前置：数据源返回 0 条  
  - 预期：显示空态文案，不报错，不影响继续其他操作  

- 场景：重复提交表单  
  - 前置：快速连续点击提交  
  - 预期：后端幂等或前端按钮禁用/防抖，最终仅一条结果生效
